# 매장 발주영역 통합 설계안

## 1. 개요

매장 관리자 발주 영역은 `매장 → 물류창고 발주 요청` 흐름을 기준으로 설계한다.  
기획서 v2.0 기준에서 매장 관리자는 `재고 조회`, `판매`, `발주 요청`, `입고 처리`, `통계 조회`를 담당하므로, 발주 기능은 판매/입고와 분리된 독립 운영 메뉴로 정리해야 한다.

이번 범위에서는 기존 매장 메뉴의 `주문 관리`를 실제 의미에 맞는 `발주 관리`로 전환하고, 판매 기능처럼 `요청 화면`, `내역 화면`, `분석 화면` 구조를 우선 잡는다.

## 2. 기획서 기준 핵심 요구사항 정리

기준 문서:

- `docs/codex/common/기획서_기준_핵심_요구사항_요약.md`
- `Stockit_기획서 v2.0.pdf`

매장 발주 영역에서 직접적으로 필요한 기능은 아래와 같다.

- 물류창고 대상 발주 요청
- 발주 상태 조회
- 발주 수정
- 발주 취소
- 상품별 발주량 통계
- 창고에서 출고된 물량의 입고 검수 연결

즉, 매장 발주는 단순 요청서 작성 화면이 아니라 `요청 → 승인/할당 → 출고 → 매장 입고`로 이어지는 라이프사이클의 출발점이어야 한다.

## 3. 필요한 화면 정리

### 3.1 발주 요청 화면

목적:

- 매장 재고 부족 SKU를 확인하고 창고로 발주 요청한다.
- 한 번의 요청에 여러 SKU를 묶어 하나의 발주건으로 저장한다.

핵심 구성:

- 좌측: 발주 대상 SKU 탐색 영역
- 우측: 발주 요청서 영역

표시/입력 항목:

- SKU 기준 상품명
- `mainCategory`, `subCategory`
- `color`, `size`
- 현재 매장 재고
- 안전재고
- 권장 발주 수량
- 요청 수량
- 요청 사유 또는 메모

동작 규칙:

- 발주 단위는 `상품`이 아니라 `SKU`다.
- 같은 SKU를 다시 담으면 새 줄이 아니라 수량 누적으로 처리한다.
- 수량은 1 이상 정수만 허용한다.
- 발주 요청 시 요청서 전체를 하나의 발주건으로 저장한다.
- `요청 완료` 이전 상태에서만 수정/취소 가능하게 설계한다.

### 3.2 발주 내역 화면

목적:

- 매장 관리자가 자신이 요청한 발주건의 진행 상태를 확인한다.
- 수정/취소 가능 상태인지 빠르게 판단한다.

목록 컬럼:

- 요청일시
- 발주번호
- 대표 상품명
- 총수량
- 총요청수량
- 진행 상태
- 도착 예정일

상세 정보:

- 발주 라인별 SKU 정보
- 요청 수량
- 승인 수량 또는 배정 수량
- 상태 이력
- 요청 메모
- 취소 사유 또는 반려 사유

### 3.3 발주 분석 화면

초기 MVP에서는 최소 뼈대만 둔다.

- 전체 발주건 수
- SKU별 누적 발주 수량
- 카테고리별 발주 비중
- 미입고 발주 현황

향후 확장 후보:

- 안전재고 미달 SKU 기반 발주 패턴
- 판매량 대비 발주량 비교
- 월별/시즌별 발주 추이
- 반복 부족 SKU 리포트

## 4. 필요한 상태 정리

기획서와 현재 코드 구조를 함께 고려할 때, 매장 발주에는 아래 상태가 필요하다.

- `DRAFT`: 작성 중인 임시 상태
- `REQUESTED`: 매장 발주 요청 완료
- `APPROVED`: 본사 또는 시스템 승인/할당 완료
- `PICKING`: 창고 피킹 진행
- `SHIPPING`: 창고 출고 완료, 매장 입고 대기
- `RECEIVED`: 매장 입고 검수 완료
- `CANCELLED`: 매장 취소
- `REJECTED`: 승인/할당 거절

초기 구현 원칙:

- UI 노출은 `요청`, `승인`, `피킹`, `배송중`, `입고완료`, `취소`, `반려` 같은 업무 용어를 사용한다.
- 내부 상태명은 영문 상수로 관리한다.
- 초기 화면에서는 `PICKING`이 아직 상세 구현되지 않더라도 상태 모델에는 선반영한다.

수정/취소 가능 기준:

- `DRAFT`, `REQUESTED`에서만 가능
- `APPROVED` 이후는 읽기 전용

입고 연결 기준:

- `SHIPPING` 상태의 발주건은 추후 `입고 관리` 화면에서 입고 검수 대상으로 보여야 한다.
- 매장 입고 확정 시 최종 상태는 `RECEIVED`로 전환하는 것이 적절하다.

## 5. 데이터 구조 초안

판매 기능이 `Sale + SaleItem[]` 구조를 사용하듯, 발주도 헤더 + 라인 구조로 분리한다.

### 5.1 StoreOrder

- `orderId`
- `storeId`
- `storeName`
- `warehouseId`
- `warehouseName`
- `requestedAt`
- `requestedBy`
- `status`
- `totalSkuCount`
- `totalRequestedQuantity`
- `memo`
- `expectedDeliveryDate`
- `statusHistory[]`

### 5.2 StoreOrderItem

- `orderId`
- `skuId`
- `productId`
- `productName`
- `mainCategory`
- `subCategory`
- `color`
- `size`
- `unitPrice`
- `currentStoreStock`
- `safetyStock`
- `recommendedQuantity`
- `requestedQuantity`
- `approvedQuantity`

설계 원칙:

- 판매처럼 SKU 스냅샷을 저장해, 이후 상품 마스터가 바뀌어도 당시 요청 맥락이 남아야 한다.
- `approvedQuantity`는 초기에는 `requestedQuantity`와 동일하게 시작해도 되지만, 추후 부분 승인 대응을 위해 필드를 둔다.

## 6. 현재 코드 기준 갭 분석

관련 파일:

- `src/config/roleMenus.js`
- `src/router/index.js`
- `src/views/store/StoreOrdersView.vue`
- `src/stores/purchaseOrder.js`
- `src/views/hq/HqPurchaseOrderView.vue`
- `src/views/hq/HqPurchaseOrderCreateView.vue`
- `src/stores/vendor.js`
- `src/stores/inventory.js`

현재 확인된 상태는 아래와 같다.

- 매장 메뉴에는 이미 `발주 관리`가 존재한다.
- 라우트 `/store/orders`도 존재한다.
- 하지만 실제 화면 `StoreOrdersView.vue`는 `준비 중인 기능입니다.` 수준의 placeholder다.
- 반대로 본사 거래처 발주 화면은 생성/수정/취소/상태 변경까지 상당 부분 구현돼 있다.
- 창고 입고 화면도 `purchaseOrder` 스토어 기준으로 `SHIPPING → COMPLETED` 흐름이 연결돼 있다.

주요 갭은 아래와 같다.

### 6.1 매장 전용 발주 데이터 모델 부재

- 현재 `src/stores/purchaseOrder.js`는 `본사 → 거래처 발주` 모델이다.
- `vendorId`, `vendorName` 중심 구조라서 `매장 → 창고 발주`를 그대로 표현하기 어렵다.
- `storeId`, `requestedByStore`, `approvedQuantity`, `receivedAt` 같은 필드가 없다.

### 6.2 도메인 불일치

- 현재 `purchaseOrder.js`와 `vendor.js` 더미 데이터는 전자/문구/생활용품 중심이다.
- 현재 `inventory.js`와 판매 기능은 의류 SKU 중심으로 재정렬돼 있다.
- 따라서 발주 요청 화면을 본사 발주 카탈로그 그대로 재사용하면 기획서 기준과 판매 기능 기준 모두와 어긋난다.

### 6.3 화면 구조 불일치

- 판매 기능은 `register/history/analysis`로 분리돼 있다.
- 발주는 아직 `/store/orders` 단일 placeholder 화면이라 정보 구조가 얕다.
- 기획서가 요구하는 `요청`, `조회`, `수정/취소`, `통계`를 한 화면에 무리하게 넣기보다 판매처럼 분리하는 편이 일관적이다.

### 6.4 입고 연결 전제 부족

- 기획서상 매장 발주 이후에는 `입고 처리`가 이어져야 한다.
- 현재 매장 `입고 관리` 화면도 placeholder라서 발주 상태와의 연결점이 아직 없다.
- 따라서 발주 상태 모델은 지금부터 `SHIPPING`, `RECEIVED`까지 고려해 설계해야 이후 입고 화면과 자연스럽게 붙는다.

### 6.5 사용자/매장 식별 정보 부족

- `auth.js`의 store 사용자 정보에는 `storeId`, `storeName`이 없다.
- 판매 스토어는 `DEFAULT_STORE_ID`를 별도로 갖고 있다.
- 발주 기능에서도 매장 식별자가 필요하므로 인증 더미 또는 별도 상수 구조를 정리해야 한다.

## 7. 발주 사이드바 구조 브레인스토밍

### 안 A

- 상위: `발주 관리`
- 하위: `발주 요청`, `발주 내역`, `발주 분석`

장점:

- 기획서의 `발주 요청`이라는 업무 용어와 가장 잘 맞는다.
- `판매 등록`과 유사하지만, 실제 행위가 `등록`보다 `요청`임을 정확히 보여준다.

### 안 B

- 상위: `발주 관리`
- 하위: `발주 등록`, `발주 현황`, `발주 통계`

장점:

- 판매 메뉴의 `등록/내역/분석` 패턴과 유사해 익숙하다.

단점:

- 매장 발주는 독립 확정이 아니라 창고 대상 요청이라 `등록`이 다소 약하다.
- `현황`은 수정/취소 가능한 과거 요청 목록이라는 의미가 덜 선명하다.

### 안 C

- 상위: `발주 관리`
- 하위: `발주 작성`, `진행 현황`, `발주 리포트`

장점:

- UX 톤은 부드럽다.

단점:

- 현재 프로젝트의 메뉴 언어와 결이 조금 다르다.
- `리포트`는 AI 리포트와 의미가 겹칠 수 있다.

### 추천안

이번 작업에서는 안 A를 추천한다.

- 상위: `발주 관리`
- 하위: `발주 요청`
- 하위: `발주 내역`
- 하위: `발주 분석`

권장 라우트:

- `/store/orders/request`
- `/store/orders/history`
- `/store/orders/analysis`
- 기존 `/store/orders`는 `/store/orders/request`로 리다이렉트

## 8. 구현 계획

### 8.1 1단계: 메뉴/라우트 구조 정리

- `roleMenus.store`의 발주 관리 children을 3개 메뉴로 확장
- `router/index.js`에 발주 요청/내역/분석 라우트 추가
- 기존 `StoreOrdersView.vue`는 분리된 `src/views/store/orders/*` 구조로 재정리

### 8.2 2단계: 매장 발주 스토어 신설

- 신규 스토어 예시: `src/stores/storeOrders.js`
- `StoreOrder + StoreOrderItem[]` 구조 도입
- localStorage 영속화
- 상태 전이 액션 정의
- 검색/상태 필터/요약 getter 정의

### 8.3 3단계: 발주 요청 화면 구현

- 의류 SKU 기반 카탈로그 구성
- 현재 재고와 안전재고를 함께 보여주는 요청 리스트 UI
- 다중 SKU 요청서 작성
- 권장 수량 표시
- 요청 확정, 요청서 비우기, 수량 수정

### 8.4 4단계: 발주 내역 화면 구현

- 상태 탭
- 발주번호/상품명/카테고리 검색
- 기간 필터
- 상세 패널
- `REQUESTED` 상태에서 수정/취소 액션

### 8.5 5단계: 발주 분석 화면 최소 구현

- 총 발주건 수
- 누적 요청 수량
- SKU별 누적 발주량 미리보기
- 카테고리 비중 요약

### 8.6 6단계: 매장 입고 연결을 위한 선반영

- `SHIPPING` 상태 데이터가 추후 `입고 관리`에서 재사용 가능하도록 필드 정리
- 필요 시 입고용 뷰 모델 함수 분리

### 8.7 7단계: 문서화 및 검증

- 구현 후 `docs/codex/store/02-implementation/매장_발주영역_구현_내역.md` 작성
- `vite build` 기준 최소 검증
- 상태 전이와 수정/취소 조건 수동 점검

## 9. 구현 시 주의사항

- 본사 거래처 발주 스토어를 그대로 재사용하지 말고, 매장 발주 도메인에 맞는 별도 스토어를 두는 편이 안전하다.
- 다만 목록/상세 UI 패턴, 상태 뱃지, 확인 모달, 히스토리 타임라인 구조는 현재 본사 발주 화면에서 재사용 가능하다.
- 현재 `vendor.js` 더미 계약 제품은 의류 도메인과 맞지 않으므로, 매장 발주 요청 화면은 우선 `inventory.js`의 SKU와 안전재고를 기준으로 만드는 것이 자연스럽다.
- 이후 본사/창고 흐름과 연결할 때는 `매장 발주`와 `거래처 발주`를 별도 엔티티로 유지해야 역할 충돌이 적다.

## 10. 테스트 기준

- 여러 SKU를 하나의 발주 요청서에 담아 한 번에 요청할 수 있어야 한다.
- 같은 SKU를 여러 번 추가하면 요청서 수량만 누적되어야 한다.
- `REQUESTED` 상태에서는 수정/취소가 가능해야 한다.
- `APPROVED` 이후에는 수정/취소가 차단되어야 한다.
- 발주 내역 목록에는 발주건 중심으로 보이고, 상세에서는 SKU 라인이 보여야 한다.
- `SHIPPING` 상태 데이터는 이후 입고 관리 화면으로 넘길 수 있는 구조여야 한다.
- 상품/카테고리/상태 기준 검색과 필터가 동작해야 한다.

## 11. 결론

매장 발주 기능은 기존 본사 거래처 발주 화면을 단순 복제하는 작업이 아니다.  
판매 기능에서 이미 맞춘 `의류 SKU`, `헤더 + 라인 저장`, `메뉴 분리`, `상태 기반 업무 흐름` 원칙을 그대로 이어서, `매장 요청 → 창고 출고 → 매장 입고` 라이프사이클의 첫 단계로 설계해야 한다.

이번 구현의 우선 목표는 아래 3가지다.

- 기획서 용어 기준으로 매장 발주 UX를 재구성
- 현재 판매/재고 도메인과 맞는 의류 SKU 기반 요청 흐름 확보
- 이후 입고/창고/본사 흐름과 연결 가능한 상태 모델 선반영
