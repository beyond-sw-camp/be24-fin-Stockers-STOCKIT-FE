# 매장 발주영역 구현 내역

## 1. 구현 목적

매장 관리자 발주 기능을 `발주 요청`, `발주 내역`, `발주 상세`, `발주 분석` 구조로 재정렬하고, 판매/재고 기능과 동일한 의류 SKU 도메인 기준으로 동작하게 만드는 것이 이번 구현의 목적이다.

핵심 구현 목표:

- 매장 발주 메뉴 구조 분리
- 매장 전용 발주 스토어 신설
- 의류 SKU 기반 발주 요청 흐름 구현
- 승인 대기 상태 기준 수정/취소 흐름 구현
- 발주 분석 확장을 위한 최소 화면 및 집계 구조 준비

## 2. 반영된 구조 변경

### 2.1 메뉴 구조

`src/config/roleMenus.js`

매장 `발주 관리` 메뉴를 다음과 같이 재구성했다.

- 발주 요청
- 발주 내역
- 발주 분석

기존 `/store/orders` 진입은 `발주 요청` 화면으로 유도되도록 정리했다.

### 2.2 라우터 구조

`src/router/index.js`

추가/변경된 라우트:

- `/store/orders/request`
- `/store/orders/request/:id/edit`
- `/store/orders/history`
- `/store/orders/history/:id`
- `/store/orders/analysis`
- 기존 `/store/orders`는 `store-order-request`로 리다이렉트

## 3. 데이터 모델 구현

### 3.1 매장 발주 스토어

`src/stores/storeOrders.js`

본사 거래처 발주 스토어와 분리된 매장 전용 발주 스토어를 추가했다.

구현 내용:

- `StoreOrder` 저장
- `StoreOrderItem[]` 저장
- 발주번호(`orderId`) 자동 생성
- `REQUESTED`, `APPROVED`, `CANCELLED` 상태 모델 적용
- 상태 이력(`statusHistory`) 저장
- localStorage 영속화
- 기본 분석용 집계 getter 제공
- `실재고`, `가용재고`, `입고예정 수량` 계산 필드 추가

헤더 필드:

- `orderId`
- `storeId`
- `storeName`
- `requestedAt`
- `requestedBy`
- `status`
- `totalSkuCount`
- `totalRequestedQuantity`
- `memo`
- `cancelReason`

라인 필드:

- `skuId`
- `productId`
- `productName`
- `mainCategory`
- `subCategory`
- `color`
- `size`
- `unitPrice`
- `currentStoreStock`
- `availableStoreStock`
- `inboundExpectedQuantity`
- `safetyStock`
- `recommendedQuantity`
- `requestedQuantity`

재고 표시 규칙:

- `currentStoreStock`: 실재고
- `inboundExpectedQuantity`: 입고예정 수량
- `availableStoreStock`: 가용재고 (`실재고 + 입고예정 수량`)

### 3.2 인증 사용자 보강

`src/stores/auth.js`

매장 발주와 이후 입고 연결을 위해 더미 인증 사용자에 아래 필드를 추가했다.

- `memberId`
- `storeId`
- `storeName`

## 4. 화면 구현 내역

### 4.1 발주 요청

`src/views/store/orders/StoreOrderRequestView.vue`

판매 등록 화면과 유사한 `좌측 검색 + 우측 요청서` 구조로 발주 요청 화면을 구현했다.

반영 기능:

- 전체 SKU 검색
- 부족/안전재고 미만 SKU 우선 정렬
- 일반 정렬 필터(`카테고리순`, `상품명순`, `재고 많은순`)
- 대분류/소분류/색상/사이즈/검색어 필터
- 실재고/가용재고 동시 노출
- 권장 발주 수량 표시
- 같은 SKU 재추가 시 요청 수량 누적
- 요청서 수량 직접 수정
- 요청 메모 입력
- 신규 요청 등록
- 기존 `REQUESTED` 발주 수정

운영 규칙:

- `매장 발주 검색`은 별도 메뉴가 아니라 `발주 요청` 화면 내부 기능으로 반영
- 자동 배치 승인은 구현하지 않고, 정책 안내 문구로만 노출
- UI 상태 문구는 `승인 대기`, `승인 완료`, `취소` 기준으로 정리

### 4.2 발주 내역

`src/views/store/orders/StoreOrderHistoryView.vue`

발주건 중심의 목록 전용 조회 화면으로 재정리했다.

반영 기능:

- 상태 탭(`전체`, `요청`, `승인 완료`, `취소`)
- 발주번호/상품명/카테고리 검색
- 기간 필터
- 정렬 옵션
- 발주건 클릭 시 상세 페이지 이동

### 4.3 발주 상세

`src/views/store/orders/StoreOrderDetailView.vue`

발주 상세 전용 페이지를 추가했다.

반영 기능:

- 발주 헤더 정보 조회
- SKU 라인별 실재고/가용재고/요청 수량 표시
- 요청 메모 / 취소 사유 / 상태 이력 조회
- `REQUESTED` 상태에서만 수정/취소 버튼 노출
- 상세 페이지 내 취소 모달 처리

### 4.4 발주 분석

`src/views/store/orders/StoreOrderAnalysisView.vue`

초기 MVP 범위에서는 최소 분석 뼈대만 구현했다.

반영 기능:

- 발주건 수
- 누적 요청 수량
- 요청 상태 건수
- 승인 완료 건수
- SKU별 누적 발주량
- 카테고리별 발주 비중

## 5. 주변 화면 정리

`src/views/store/StoreDashboardView.vue`

- 매장 대시보드의 발주 관리 바로가기를 `/store/orders/request`로 변경

## 6. 구현 시 적용한 원칙

- 본사 거래처 발주 스토어는 재사용하지 않고 매장 전용 스토어를 별도 구성
- 데이터 도메인은 현재 판매/재고와 같은 의류 SKU 기준으로 통일
- 수정/취소는 `REQUESTED`(승인 대기) 상태에서만 허용
- 자동 승인 스케줄러는 제외하고 정책 안내만 반영
- 가용재고는 `실재고 + 입고예정 수량` 기준으로 계산
